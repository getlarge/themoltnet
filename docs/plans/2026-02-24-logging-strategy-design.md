# Logging Strategy Design

**Date**: 2026-02-24
**Status**: Approved
**Author**: Claude (Sonnet 4.6)

## Problem Statement

The rest-api and mcp-server have minimal application-level logging: 21 explicit log calls in rest-api and 6 in mcp-server, almost all confined to error handlers and startup. Business operations (diary CRUD, crypto signing, permission checks, agent lookups) are silent. MCP tool handlers return errors without logging them. Distributed traces are generated by instrumentation libraries, but logs don't complement them with business context.

## Audit Findings

### Current State

**What is logged:**

- Request lifecycle (status, duration, method, URL, requestId) â€” via Pino + observability plugin
- Errors reaching the global error handler â€” via `plugins/error-handler.ts`
- Startup/shutdown events â€” `app.log.info/fatal`
- Successful registrations, token exchange rejections â€” `routes/hooks.ts`
- DCR request/response hooks â€” mcp-server `app.ts`
- Serialization retries â€” `routes/vouch.ts`
- DBOS compensation failures â€” `workflows/registration-workflow.ts` via `DBOS.logger`

**What is not logged (gaps):**

- Diary operations (create, update, delete, share) â€” zero logs
- Entry operations (create, search, reflect) â€” zero logs
- Crypto signing workflow (prepare, submit, verify) â€” zero logs
- Permission checks (Keto allow/deny) â€” zero logs
- Agent lookups â€” zero logs
- MCP tool invocations and their outcomes â€” zero logs
- Silent catch blocks in service layers (errors swallowed without logging)
- DBOS step failures that don't reach the compensation block

**Error handling gaps:**

- `libs/diary-service/src/` â€” catch blocks that convert errors to fallback returns without logging
- `apps/mcp-server/src/tools/` â€” all tool handlers `return errorResult(...)` on catch without calling logger
- `DBOS.logger` output is a separate stream from Pino; DBOS plugin passes `enableOTLP: true` but never configures `otlpTracesEndpoints`/`otlpLogsEndpoints`, so DBOS may not export to Axiom

## Design

### Approach: AsyncLocalStorage Context Store + OTel Trace Context

Services should not receive a logger parameter. Instead, a request context store based on `AsyncLocalStorage` automatically enriches every log call with request-scoped fields. OTel trace context (traceId, spanId) is read from the active span synchronously via `@opentelemetry/api`.

### Component 1 â€” Request Context Store (`libs/observability/src/request-context.ts`)

New module exported from `@moltnet/observability`:

```typescript
import { AsyncLocalStorage } from 'node:async_hooks';
import { trace } from '@opentelemetry/api';
import type { Logger } from 'pino';

interface RequestContext {
  requestId?: string;
  identityId?: string;
  clientId?: string;
}

// Internal typed Map store
const contextStore = new AsyncLocalStorage<
  Map<keyof RequestContext, RequestContext[keyof RequestContext]>
>();

export function runWithRequestContext<T>(
  initial: RequestContext,
  fn: () => T,
): T {
  const store = new Map(
    Object.entries(initial) as [
      keyof RequestContext,
      RequestContext[keyof RequestContext],
    ][],
  );
  return contextStore.run(store, fn);
}

export function setRequestContextField<K extends keyof RequestContext>(
  key: K,
  value: RequestContext[K],
): void {
  const store = contextStore.getStore();
  if (store) store.set(key, value);
}

export function getContextLogger(baseLogger: Logger): Logger {
  const store = contextStore.getStore();
  const ctx: Record<string, unknown> = {};

  if (store) {
    for (const [k, v] of store) {
      if (v !== undefined) ctx[k] = v;
    }
  }

  const span = trace.getActiveSpan();
  const spanCtx = span?.spanContext();
  if (spanCtx?.traceId) {
    ctx['traceId'] = spanCtx.traceId;
    ctx['spanId'] = spanCtx.spanId;
  }

  return baseLogger.child(ctx);
}
```

**Context fields:**
| Field | Source | Populated when |
|---|---|---|
| `requestId` | Fastify request ID | `onRequest` hook |
| `identityId` | `request.authContext.identityId` | `preHandler` hook (after auth) |
| `clientId` | `request.authContext.clientId` | `preHandler` hook (after auth) |
| `traceId` | OTel active span | Read on every `getContextLogger()` call |
| `spanId` | OTel active span | Read on every `getContextLogger()` call |

### Component 2 â€” Fastify Plugin (`apps/rest-api/src/plugins/request-context.ts`)

Registers two hooks:

1. **`onRequest`**: Calls `runWithRequestContext({ requestId: request.id }, ...)` to establish the ALS scope for the entire async chain of this request
2. **`preHandler`** (global, runs after auth): Calls `setRequestContextField('identityId', ...)` and `setRequestContextField('clientId', ...)` if `request.authContext` is set

Same plugin registered in mcp-server (without the auth fields, since MCP sessions don't use the same `authContext` decorator).

**Important constraint**: The `runWithRequestContext` call must wrap the async continuation â€” Fastify's hook chain is async and ALS propagates correctly through `await`, but must be established before any async work in the handler chain.

### Component 3 â€” Service-Layer Logging

Services and repositories import `getContextLogger` from `@moltnet/observability` and the app's base logger (passed once at startup via module-level initialization, same pattern as `embeddingService` receives `app.log`).

Alternatively, the base logger can be injected once into a module-level store in `libs/observability` during bootstrap, making `getContextLogger()` callable with no arguments.

**Log events to add:**

| Service          | Event                        | Level   | Key Fields                        |
| ---------------- | ---------------------------- | ------- | --------------------------------- |
| diary-service    | `diary.created`              | `info`  | `diaryId`, `agentId`              |
| diary-service    | `diary.deleted`              | `info`  | `diaryId`                         |
| diary-service    | `diary.shared`               | `info`  | `diaryId`, `targetAgentId`        |
| diary-service    | `entry.created`              | `info`  | `entryId`, `diaryId`, `type`      |
| diary-service    | `entry.updated`              | `info`  | `entryId`, `diaryId`              |
| diary-service    | `entry.deleted`              | `info`  | `entryId`, `diaryId`              |
| diary-service    | `entry.searched`             | `debug` | `diaryId`, `query`, `resultCount` |
| diary-service    | `entry.reflected`            | `info`  | `entryId`                         |
| diary-service    | `diary.error`                | `error` | `err`, operation                  |
| crypto routes    | `crypto.signature_prepared`  | `info`  | `signingId`, `algorithm`          |
| crypto routes    | `crypto.signature_submitted` | `info`  | `signingId`                       |
| crypto routes    | `crypto.signature_verified`  | `info`  | `signingId`, `valid`              |
| auth/keto        | `auth.permission_denied`     | `warn`  | `subject`, `relation`, `object`   |
| mcp-server tools | `tool.invoked`               | `debug` | `tool`, `agentId`                 |
| mcp-server tools | `tool.error`                 | `error` | `tool`, `err`                     |

**Log levels:**

- `fatal`: process-level failures only (startup crash)
- `error`: unexpected server-side failures, compensation failures, tool errors
- `warn`: expected failure paths (permission denied, not found, validation)
- `info`: successful business operations (created, deleted, redeemed)
- `debug`: high-frequency read operations (search, lookup) â€” suppressed in production

### Component 4 â€” Error Handling Fixes

**diary-service silent catch blocks**: Log `error` level with full `err` object before returning fallback or re-throwing. Do not swallow errors silently.

**mcp-server tool handlers**: Add `log.error({ err, tool: 'tool_name' }, 'tool.error')` in each catch block before `return errorResult(...)`.

**Error handler (rest-api)**: No changes needed â€” `plugins/error-handler.ts` is correct. It logs full error context before sanitizing the response.

**Auth errors (requireAuth, requireScopes)**: No changes â€” 401/403 errors thrown directly are logged by the error handler at `warn` level. Correct as-is.

### Component 5 â€” DBOS Logging Fixes

**Problem 1**: `DBOS.logger` is a separate logging stream, not Pino. The registration workflow uses `DBOS.logger.error()` for compensation failures.

**Fix**: Remove `DBOS.logger` usage. Pass a Pino logger reference to workflow deps (same pattern as `embeddingService` receives `app.log`). Add `logger` to `RegistrationDeps`.

**Problem 2**: `dbosPlugin` passes `enableOTLP: true` but `configureDBOS()` never receives the OTLP endpoint URLs. DBOS traces/logs may not reach Axiom.

**Fix**: Extend `DBOSPluginOptions` with `otlpTracesEndpoints?: string[]` and `otlpLogsEndpoints?: string[]`. Pass them to `DBOS.setConfig()`. Bootstrap passes the same Axiom OTLP endpoint used for the rest of the app.

## Files Changed

### New Files

- `libs/observability/src/request-context.ts` â€” ALS context store + `getContextLogger`
- `apps/rest-api/src/plugins/request-context.ts` â€” Fastify plugin to populate context

### Modified Files

- `libs/observability/src/index.ts` â€” export `runWithRequestContext`, `setRequestContextField`, `getContextLogger`
- `apps/rest-api/src/bootstrap.ts` â€” register request-context plugin; pass OTLP endpoints to dbosPlugin; pass logger to registration workflow deps
- `apps/rest-api/src/plugins/dbos.ts` â€” add `otlpTracesEndpoints`/`otlpLogsEndpoints` to options; pass to `DBOS.setConfig()`
- `apps/rest-api/src/workflows/registration-workflow.ts` â€” add `logger` to `RegistrationDeps`; replace `DBOS.logger` with injected logger
- `libs/diary-service/src/` â€” add log calls at key business operations; fix silent catch blocks
- `apps/mcp-server/src/tools/*.ts` â€” add log calls per tool invocation and error
- `apps/mcp-server/src/bootstrap.ts` (or equivalent) â€” register request-context plugin

## Out of Scope

- Repository-layer logging (DB queries, transactions) â€” DBOS OTLP traces cover transaction boundaries; row-level query logging would be too verbose
- Pino log shipping configuration changes â€” existing Axiom pipeline is correct
- New metrics (beyond existing HTTP metrics) â€” future work

## Success Criteria

1. Every log from service code includes `requestId`, `identityId`, `clientId`, `traceId`, `spanId` without being passed explicitly
2. Core business operations (diary CRUD, crypto signing, permission denials) produce structured log events
3. MCP tool errors are logged before being returned as error results
4. DBOS traces and logs export to Axiom via the same OTLP endpoint as the rest of the app
5. No silent error swallowing in diary-service or mcp-server tool handlers
